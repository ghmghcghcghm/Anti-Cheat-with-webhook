Setup & Installation
Get the system running in your game in 5 minutes

Step 1 — Enable HttpService

Discord webhooks require HttpService to be enabled. This is disabled by default for security.

1

Open Roblox Studio

Launch Studio and open your game project.

2

Game Settings → Security

Click Home in the top ribbon → Game Settings → Security tab.

3

Enable "Allow HTTP Requests"

Toggle the switch to ON. Click Save. HttpService:PostAsync() will now work at runtime.

⚠️
Security note: HttpService only works in live game servers, not in Studio Play mode unless you're testing with a team-test server. HTTP calls are silently blocked in single-player Studio testing.

Step 2 — Create the Discord Webhook

1

Open your Discord server

Navigate to the text channel where you want alerts (e.g., #anti-cheat-logs).

2

Edit Channel → Integrations

Click the ⚙️ gear icon beside the channel name → Integrations → Webhooks.

3

Create New Webhook

Click New Webhook. Name it "AntiCheat Bot". Optionally set an avatar image.

4

Copy Webhook URL

Click Copy Webhook URL. It looks like: https://discord.com/api/webhooks/12345.../AbCdEf...

5

Paste into Config

In MainAntiCheatServer.server.lua, find the CONFIG table and replace the WebhookURL value.

lua — MainAntiCheatServer.server.lua 
local CONFIG = {
    -- ... other settings ...

    -- 🔑 Paste your webhook URL here:
    WebhookURL     = "https://discord.com/api/webhooks/YOUR_ID/YOUR_TOKEN",
    WebhookEnabled = true,
}

Step 3 — Place Scripts in Studio

1

Create Modules folder

In the Explorer, right-click ReplicatedStorage → Insert Object → Folder. Name it Modules.

2

Add ModuleScripts

Inside Modules, create 5 ModuleScript objects. Name and paste the code for each from the provided files.

3

Add Server Script

In ServerScriptService, create a Script. Paste MainAntiCheatServer.server.lua content.

4

Add LocalScript

In StarterPlayer → StarterPlayerScripts, create a LocalScript. Paste AntiCheatClient.client.lua content.

5

Test in Play Mode

Press Play. You should see [AntiCheat] System started in the Output window.

✅
The system is ready. Try the admin command !replay YourUsername thirdperson in chat to test the replay system on yourself.

Class Responsibilities

AntiCheatManager

The top-level orchestrator. Creates one PlayerTracker per player, runs the heartbeat loop, handles flagging logic (scoring threshold), freezes characters, and coordinates webhook delivery. Exposes OnPlayerFlagged callback for external scripts.

PlayerTracker

One instance per player. Maintains a rolling state: last position, air time, velocity. Runs three detection checks every tick: speed, fly, and teleport. Also holds the player's ReplayBuffer and accepts camera direction updates from the client RemoteEvent.

ReplayBuffer

A pre-allocated circular ring buffer (no GC pressure). Stores the last N×tickRate frames. O(1) push, O(N) ordered retrieval. Frame data includes position, CFrame, velocity, humanoid state, and camera look vector.

ReplayController

Spawns a visible dummy character in the world. Steps through frames, repositioning the dummy and firing AC_ReplayEvent to the viewer's client each tick. Supports three camera modes via the client-side listener. Cleans up dummy on completion.

WebhookService

Formats Discord embed payloads (with colour-coded severity), encodes JSON, and POSTs via HttpService:PostAsync() on a spawned thread so it never blocks the game loop. Handles encode/send errors gracefully with pcall.

Detection Logic
How each exploit type is caught server-side

Speed Detection

Every Heartbeat, the tracker calculates distance / elapsed_time between the character's current and previous RootPart positions. If this exceeds MaxSpeedStuds × 1.25 (the 1.25× buffer absorbs physics jitter), a speed violation fires.

lua — PlayerTracker.lua 
function PlayerTracker:_checkSpeed(currentPos, dt)
    local dist  = (currentPos - self._lastPosition).Magnitude
    local speed = dist / dt  -- studs per second

    if speed > (self.Config.MaxSpeedStuds * 1.25) then
        self:_addViolation("Speed", self.Config.SpeedViolationScore, ...)
    end
end

Fly Detection

An air time accumulator ticks up while the humanoid is not in a grounded state and not in a normal jump arc (Jumping/Freefall states are excluded). When _airTime exceeds MaxAirTimeSecs, a fly violation fires and the accumulator is partially reset (to 50% of threshold) to prevent spam.

lua — PlayerTracker.lua 
local HUMANOID_STATES_ON_GROUND = {
    [Enum.HumanoidStateType.Running]  = true,
    [Enum.HumanoidStateType.Climbing] = true,
    [Enum.HumanoidStateType.Swimming] = true,
    -- ... etc
}

function PlayerTracker:_checkFly(currentState, dt)
    local isInAir = not HUMANOID_STATES_ON_GROUND[currentState]
        and currentState ~= Enum.HumanoidStateType.Jumping
        and currentState ~= Enum.HumanoidStateType.Freefall

    if isInAir then
        self._airTime = self._airTime + dt
    else
        self._airTime = math.max(0, self._airTime - dt * 2)
    end

    if self._airTime > self.Config.MaxAirTimeSecs then
        self:_addViolation("Fly", self.Config.FlyViolationScore, ...)
        self._airTime = self.Config.MaxAirTimeSecs * 0.5
    end
end

Teleport Detection

If the distance between two consecutive frames exceeds TeleportThreshold studs, a teleport violation fires. The threshold defaults to 80 studs — large enough to never fire for normal movement or lag-compensation, but catches hard teleports immediately.

Violation Scoring

Violations accumulate into a score. The score decays passively at ViolationDecayRate pts/second (default 2 pts/s). This means:

  • A single brief speed hiccup (+15) decays away in ~7.5 seconds without flagging.
  • Sustained speed hacking fires repeatedly, building score faster than decay.
  • Multiple exploit types stack to cross the flag threshold faster.
ℹ️
Frozen players (WalkSpeed = 0) are skipped by all detection checks, preventing false positives after flagging.
Replay System
Circular buffer + dummy reconstruction + 3 camera modes

Replay Buffer

Each PlayerTracker owns a ReplayBuffer — a pre-allocated fixed-size array using a write-head pointer. No table.insert/remove = no GC pressure on a hot path.

lua — Frame Data Structure 
{
    timestamp     : number,    -- tick() value
    position      : Vector3,   -- world position
    cframe        : CFrame,    -- full rotation + position
    velocity      : Vector3,   -- AssemblyLinearVelocity
    humanoidState : Enum,      -- HumanoidStateType
    lookVector    : Vector3,   -- from client camera (non-critical)
}

At 20 ticks/second × 30 seconds = 600 frames max per player. The circular buffer overwrites the oldest frame when full — always keeping the most recent 30 seconds.

Violation Frame Highlighting

When a violation was detected during the recording window, the server can pass violationFrameIdx to ReplayController:PlayReplay(). At that frame, the dummy's body turns red, a SelectionBox highlights it, and the client HUD shows a "⚠️ VIOLATION FRAME DETECTED" popup for 2 seconds.

Player Hiding During Replay

The client script sets LocalTransparencyModifier = 1 on all other players' parts when replay starts. This is purely local (visual only) — the server is unaffected. Parts are restored to 0 when replay ends.

Configuration Reference
All adjustable values in the CONFIG table
lua — Full CONFIG table 
local CONFIG = {
    -- ── Detection ────────────────────────────────────────────────
    MaxSpeedStuds          = 32,    -- studs/s; default WalkSpeed×2
    MaxAirTimeSecs         = 3.5,   -- seconds before fly triggers
    TeleportThreshold      = 80,    -- stud jump = teleport

    -- ── Scoring ──────────────────────────────────────────────────
    SpeedViolationScore    = 15,
    FlyViolationScore      = 20,
    TeleportViolationScore = 30,
    ViolationDecayRate     = 2,     -- pts/second passive decay
    FlagThreshold          = 60,    -- score to freeze + report

    -- ── Cooldown ─────────────────────────────────────────────────
    ReportCooldownSecs     = 30,    -- min seconds between webhook fires

    -- ── Replay ───────────────────────────────────────────────────
    BufferLengthSecs       = 30,    -- seconds of footage stored
    TickRate               = 20,    -- frames/second recorded

    -- ── Webhook ──────────────────────────────────────────────────
    WebhookURL     = "https://discord.com/api/webhooks/...",
    WebhookEnabled = true,
}

Tuning Tips

  • Reduce false positives: Increase MaxSpeedStuds if your game has speed boosts, or increase MaxAirTimeSecs for games with long jumps.
  • Increase sensitivity: Lower FlagThreshold for stricter enforcement, or raise ViolationDecayRate to make detection more forgiving.
  • Reduce webhook spam: Increase ReportCooldownSecs for high-traffic servers.
  • Memory budget: BufferLengthSecs × TickRate = frame count per player. 30×20 = 600 frames. Each frame holds ~6 values.

Security Notes
Design decisions that prevent bypass
✅ SECURE
  • All detection is server-side only
  • Character freeze is server-applied
  • Client only sends camera direction
  • RemoteEvent inputs are validated + rate-limited
  • Webhook calls are server-only (never client)
  • pcall wraps all HTTP operations
⚠️ LIMITATIONS
  • Cannot catch client-side visuals (walls, ESP)
  • Lag spikes may trigger false teleport flags
  • Very slow speed hacks may slip under threshold
  • Does not detect aim-bot (no bullet tracking)

RemoteEvent Abuse Prevention

The AC_CameraDirection RemoteEvent is rate-limited to one update per 0.4 seconds per player. Each incoming value is validated:

  • typeof(lookVector) == "Vector3" — must be a Vector3
  • lookVector.Magnitude > 0.01 — must not be zero
  • lookVector.Magnitude < 10 — must not be an abnormally large vector

Camera direction is used only for replay reconstruction, never for detection. A malicious client sending fake camera data has zero impact on whether they are detected.

Webhook URL Security

The webhook URL is stored only in a server Script inside ServerScriptService. It is never loaded into ReplicatedStorage or LocalScripts. Clients cannot access it via the Roblox API. Keep it out of public GitHub repositories.

⚠️
Recommended for production: Store the webhook URL in a ServerStorage StringValue or a DataStore entry instead of hardcoding it in the script. This allows you to rotate it without publishing a new script.
